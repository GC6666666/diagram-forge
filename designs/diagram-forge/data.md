# Data Design: Diagram Forge

## Overview

Stateless API with ephemeral storage. No persistent user data. No database (beyond SQLite for API keys in v1). All workflow artifacts expire after 24 hours. Users receive download links and output is streamed/discarded immediately after delivery.

---

## Storage Architecture

### Directory Structure

```
$DF_DATA_ROOT/            # set via DF_DATA_ROOT env var (default: /var/diagram-forge)
├── input/                # transient input artifacts
│   ├── {job_id}/         # one subdir per job
│   │   ├── image.ext     # uploaded image (original)
│   │   ├── audio.wav     # uploaded audio (converted from MP3)
│   │   └── text.txt      # text input (written to file for pipeline)
│   └── _staging/         # temp files during upload (cleaned aggressively)
├── output/               # generated diagram files (served to user)
│   ├── {job_id}/
│   │   ├── diagram.excalidraw.json
│   │   ├── diagram.drawio.xml
│   │   └── diagram.svg
├── tmp/                  # intermediate pipeline artifacts
│   ├── {job_id}/
│   │   ├── whisper_transcript.txt  # speech → text output
│   │   ├── ai_prompt.txt           # sent-to-Claude prompt (for debugging)
│   │   ├── ai_raw_response.json    # raw Claude output (before parsing)
│   │   └── ai_parsed.json          # parsed into internal format
├── logs/                  # structured JSON logs
│   ├── app.log            # application logs (rotated daily)
│   └── access.log         # request/response access logs
├── state/                 # minimal runtime state
│   └── jobs.json          # in-memory job tracker (written to disk periodically)
└── cleanup.lock           # lockfile for TTL cleanup process
```

### Storage Rules

- All paths under `$DF_DATA_ROOT` — no system `/tmp`, no `$HOME`
- `$DF_DATA_ROOT` must be on a filesystem with at least 5GB free
- Input files deleted immediately after pipeline consumption
- Output files deleted after TTL expiry (24h) — user must download in time
- Each job gets a UUID (`job_id = uuid4()`) generated at request receipt
- Directory permissions: `0700` (owner only)

### Filesystem Lifecycle

| Path | TTL | Cleanup Trigger |
|---|---|---|
| `input/{job_id}/` | 0 (immediate) | After pipeline reads input |
| `output/{job_id}/` | 24h | Background task or lazy on access |
| `tmp/{job_id}/` | 0 (immediate) | After pipeline completes |
| `logs/*.log` | 7 days | Logrotate (daily, keep 7) |

## Data Models

### Job Record (In-Memory + Persisted)

```python
class JobStatus(StrEnum):
    RECEIVED = "received"
    PROCESSING = "processing"
    COMPLETED = "completed"
    FAILED = "failed"

@dataclass(frozen=True)
class Job:
    id: str                          # UUID4
    status: JobStatus
    input_type: Literal["text", "image", "audio"]
    diagram_type: Literal["architecture", "sequence", "flowchart"]
    output_format: Literal["excalidraw", "drawio", "svg"]
    created_at: datetime             # UTC
    completed_at: datetime | None
    error_code: str | None
    error_message: str | None
    # Timing
    whisper_duration_ms: int | None  # 0 if not applicable
    ai_generation_duration_ms: int
    export_duration_ms: int
    total_duration_ms: int
    # Size tracking (for limits)
    input_size_bytes: int
    output_size_bytes: int | None
```

Job records are held in-memory during processing. On state transitions (`processing` → `completed`/`failed`), the job is written to `state/jobs.jsonl` (append-only log) for observability and TTL cleanup coordination.

### Internal Diagram Representation (Minimal v1)

> **Deferred full design.** PRD Q4 explicitly defers this. For v1, the pipeline is:
> Input → AI Generation (prompt) → Excalidraw JSON → Exporter → other formats.
>
> The minimal internal representation is the Excalidraw JSON itself, converted directly to Draw.io XML and SVG via dedicated exporters. This avoids a separate canonical model until the prompt strategy is validated.

For v1, the internal representation IS the Excalidraw JSON. Future phases will introduce a canonical model if/when format conversion requires it.

## Schema Definitions

### Excalidraw JSON Schema

Source: `packages/excalidraw/index.tsx` (Excalidraw v0.18+) and `packages/excalidraw/types.ts`

Diagram Forge generates and emits the `ExcalidrawData` structure:

```typescript
// Simplified from @excalidraw/excalidraw types (v0.18+)
interface ExcalidrawData {
  type: "excalidraw";
  version: number;        // Excalidraw schema version
  source: "diagram-forge";
  elements: readonly ExcalidrawElement[];
  appState: ExcalidrawAppState;
  files?: BinaryFiles;
}
```

**Key ExcalidrawElement types (generated by Diagram Forge):**

```typescript
// All elements share _ExcalidrawElementBase
type _ExcalidrawElementBase = Readonly<{
  id: string;                           // UUID
  x: number; y: number;                 // canvas coordinates
  width: number; height: number;
  strokeColor: string;                  // hex, e.g. "#1e1e1e"
  backgroundColor: string;              // hex, e.g. "#fafafa"
  fillStyle: FillStyle;                // "hachure" | "cross-hatch" | "solid" | "zigzag"
  strokeWidth: number;                 // px
  strokeStyle: StrokeStyle;             // "solid" | "dashed" | "dotted"
  strokeStyle: StrokeStyle;             // "solid" | "dashed" | "dotted"
  roundness: null | { type: RoundnessType; value?: number };
  roughness: number;                   // 0-2, roughjs parameter
  opacity: number;                      // 0-100
  angle: number;                        // radians
  seed: number;                         // random seed for roughjs
  version: number;                      // monotonic counter
  versionNonce: number;
  index: FractionalIndex | null;        // ordering key
  isDeleted: boolean;
  groupIds: readonly GroupId[];
  frameId: string | null;
  boundElements: readonly BoundElement[] | null;
  updated: number;                     // epoch ms
  link: string | null;
  locked: boolean;
}>;

// Concrete element types used by Diagram Forge v1:
type ExcalidrawRectangleElement = _ExcalidrawElementBase & { type: "rectangle" };
type ExcalidrawDiamondElement   = _ExcalidrawElementBase & { type: "diamond" };
type ExcalidrawEllipseElement    = _ExcalidrawElementBase & { type: "ellipse" };
type ExcalidrawTextElement       = _ExcalidrawElementBase & {
  type: "text";
  fontSize: number;
  fontFamily: FontFamilyValues;
  text: string;
  textAlign: TextAlign;
  verticalAlign: VerticalAlign;
  autoResize: boolean;
  lineHeight: number;
};
type ExcalidrawArrowElement = _ExcalidrawElementBase & {
  type: "arrow";
  points: readonly { x: number; y: number }[];
  lastCommittedPoint: { x: number; y: number } | null;
  startBinding: ArrowheadBinding | null;
  endBinding: ArrowheadBinding | null;
};
type ExcalidrawLinearElement = _ExcalidrawElementBase & {
  type: "line";
  points: readonly { x: number; y: number }[];
  lastCommittedPoint: { x: number; y: number } | null;
};
```

**AppState (minimal, for export):**

```typescript
interface ExcalidrawAppState {
  viewBackgroundColor: string;
  zenModeEnabled: boolean;
  viewModeEnabled: boolean;
}
```

**File attachments (BinaryFiles):** v1 does not embed images in the Excalidraw JSON — input images are consumed by the AI layer, not embedded in the output. `files` field is `undefined` in v1.

### Draw.io XML Schema

Source: mxGraph (draw.io engine) — `javascript/examples/editors/diagrams/*.xml`

Diagram Forge generates the `mxGraphModel` XML format:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<mxGraphModel>
  <root>
    <!-- Root diagram node -->
    <Diagram label="DiagramForge Output" id="0">
      <mxCell />
    </Diagram>
    <!-- Default layer -->
    <Layer label="Default Layer" id="1">
      <mxCell parent="0" />
    </Layer>
    <!-- Shape nodes -->
    <Shape label="Node Label" id="2">
      <mxCell style="rounded" vertex="1" parent="1">
        <mxGeometry x="100" y="80" width="120" height="60" as="geometry" />
      </mxCell>
    </Shape>
    <!-- Arrow edges -->
    <Edge label="Arrow Label" id="3">
      <mxCell edge="1" parent="1" source="2" target="4">
        <mxGeometry as="geometry">
          <mxPoint x="220" y="110" as="sourcePoint" />
          <mxPoint x="340" y="110" as="targetPoint" />
          <Array as="points" />
        </mxGeometry>
      </mxCell>
    </Edge>
  </root>
</mxGraphModel>
```

**Key element types:**

| mxGraph type | Used for |
|---|---|
| `<Diagram>` | Root node (id="0"), contains `<mxCell />` |
| `<Layer>` | Default layer (id="1"), parent="0" |
| `<mxCell vertex="1">` | Rectangle, diamond, ellipse shapes |
| `<mxCell edge="1">` | Arrows and connectors |
| `<mxGeometry as="geometry">` | Position (x, y) and size (width, height) |
| `<mxPoint as="sourcePoint"/>` | Arrow start coordinate |
| `<mxPoint as="targetPoint"/>` | Arrow end coordinate |
| `<Array as="points">` | Intermediate waypoints for bent arrows |
| `<Shape label="...">` | Semantic wrapper for shapes |
| `<Edge label="...">` | Semantic wrapper for arrows |

**Style attributes** (`style="..."` on mxCell):
- `rounded` — rectangle with rounded corners
- `ellipse` — oval/circle
- `rhombus` — diamond (decision node)
- `swimlane` — horizontal divider (sequence diagrams)
- `fillColor=#hex` — background color
- `strokeColor=#hex` — border color
- `dashed=1` — dashed line style
- `verticalEdge` — vertical routing for sequence diagrams

### SVG Output Schema

v1 produces **rendered SVG** (single vector path tree), not **editable SVG** (grouped semantic elements). This is the simpler path — the Excalidraw-to-SVG exporter from `@excalidraw/utils` handles this natively:

```python
from excalidraw_utils import export_to_svg

svg_bytes: bytes = export_to_svg(elements=elements, app_state=app_state)
```

Rendered SVG: all shapes become `<path>` elements with no semantic grouping. This satisfies the "download SVG" requirement without the full editable-SVG engineering effort. Decision deferred per PRD open question Q9.

## Cleanup / TTL Strategy

### Cleanup Process

A background cleanup process runs every 10 minutes via `cron` or a lightweight async task within the FastAPI app:

```bash
# Cleanup cron entry (or scheduled task)
*/10 * * * * find /var/diagram-forge/output -type d -mmin +1440 -exec rm -rf {} + 2>/dev/null
```

**Implementation approach:**

1. **Lazy cleanup on access** (preferred): When a user requests `/jobs/{id}/download/{format}`, check the file's `mtime`. If > 24h, return `410 Gone` instead of serving. This avoids the need for a dedicated cleanup process for the output directory.

2. **Aggressive background cleanup** (output dir only): A `httpx` background task or asyncio `PeriodicTimer` runs every 10 minutes:
   ```python
   async def cleanup_expired_outputs():
       cutoff = time.time() - 24 * 3600
       for job_dir in Path(OUTPUT_DIR).iterdir():
           if job_dir.stat().st_mtime < cutoff:
               shutil.rmtree(job_dir)
   ```

3. **Input directory**: Files are deleted immediately after the pipeline reads them. The input directory should rarely contain anything older than seconds.

4. **Tmp directory**: Deleted immediately after the job completes (success or failure).

5. **State directory**: `jobs.jsonl` is append-only. A separate compaction process truncates the log to the last N entries or entries within the TTL window.

### TTL Configuration

| Constant | Value | Env Var |
|---|---|---|
| `OUTPUT_TTL_SECONDS` | 86400 (24h) | `DF_OUTPUT_TTL_SECONDS` |
| `LOG_RETENTION_DAYS` | 7 | `DF_LOG_RETENTION_DAYS` |
| `MAX_INPUT_SIZE_BYTES` | ~10MB (image), ~10MB (audio) | `DF_MAX_INPUT_SIZE_BYTES` |
| `MAX_TEXT_CHARS` | 4000 | `DF_MAX_TEXT_CHARS` |

## API Key Management

### v1: Environment Variable (Simplest)

```bash
# Single API key for v1
export DF_API_KEY="df_v1_$(openssl rand -hex 32)"

# Multiple keys (v1.1): comma-separated list
export DF_API_KEYS="df_v1_key1,df_v1_key2,df_v1_key3"
```

### v1.1: JSON Config File (Multiple Keys + Metadata)

```
$DF_CONFIG_DIR/api_keys.json
```

```json
{
  "version": 1,
  "keys": [
    {
      "key": "df_v1_...",
      "name": "alice-laptop",
      "created_at": "2026-03-22T00:00:00Z",
      "rate_limit_rpm": 10,
      "enabled": true
    }
  ],
  "default_rate_limit_rpm": 5
}
```

**Validation logic:**
- If `$DF_API_KEY` env var is set, use it (single-key mode, no config file needed)
- If `$DF_CONFIG_DIR/api_keys.json` exists, load keys from it (multi-key mode)
- If neither exists, refuse to start (fail fast with clear error)
- Keys are never logged, never included in responses, shown only once at creation time

### v2+ (Out of Scope for v1)

- SQLite-backed key store with CRUD API
- Per-key rate limits and usage tracking
- Key rotation

## Implementation Notes

### Input Validation (System Boundary)

All inputs validated before any processing:

```
image:   mime_type in {png, jpeg, webp}
          size <= 10MB (checked on raw bytes)
          dimensions <= 4096x4096 (checked via PIL after read)
audio:   mime_type in {mp3, wav, mpeg}
          size <= 10MB
          duration <= 60s (checked via librosa after read)
text:    len(text) <= 4000 chars
          no null bytes
          encoding: utf-8
```

Any validation failure returns HTTP 422 with a descriptive `error_code`.

### Excalidraw JSON Generation Strategy

Claude returns text. We validate and parse it:

1. Claude is prompted to return Excalidraw JSON as a markdown code block
2. We extract the code block content
3. We validate it against the Excalidraw element schema
4. If invalid, attempt repair (patch common issues) or return error with `AI_GENERATION_FAILED`
5. We write the validated JSON to `output/{job_id}/diagram.excalidraw.json`

The prompt instructs Claude to produce a restricted subset of elements (rectangle, diamond, ellipse, text, arrow, line) with consistent styling to minimize schema mismatches.

### Draw.io XML Generation Strategy

Generated by transforming the validated Excalidraw JSON:

```
ExcalidrawElement → mxCell XML
- rectangle/diamond/ellipse → <mxCell vertex="1">
- arrow/line → <mxCell edge="1">
- text → <mxCell vertex="1"> with HTML label
```

Layout: Claude specifies x/y/width/height in the Excalidraw JSON. These map directly to `mxGeometry` attributes. No layout algorithm needed — Claude generates coordinates.

### SVG Generation Strategy

Use `@excalidraw/utils` `export_to_svg()` directly on the Excalidraw elements. No intermediate transformation.

### File Naming Conventions

```
# Input directory
{job_id}/
  image.png          # normalized from any supported format
  audio.wav          # normalized to WAV via ffmpeg
  text.txt           # written as-is

# Output directory
```
